# UMAP（Uniform Manifold Approximation and Projection）

> **一言でいうと**：「高次元データの構造を壊さずに、2〜3次元に圧縮して可視化する技術」

---

## なぜ次元削減が必要か？——次元の呪い

テキストをベクトル化すると、数百〜数千次元の空間にデータが存在します。たとえば `text-embedding-3-small` なら **1536次元**。

この高次元空間には、直感に反する問題が起きます。これを**次元の呪い（Curse of Dimensionality）**といいます。

### 主な問題点

| 問題 | 内容 |
|------|------|
| 距離が意味を失う | 次元が増えると、どの点も互いにほぼ同じ距離になってしまう |
| データが「スカスカ」になる | 空間の体積が爆発的に増えるため、データが薄く散らばる |
| 計算コストが爆発 | クラスタリングやk-NN探索がどんどん重くなる |

```
2次元：○ △ ◇ が近い・遠いと判断できる
1536次元：全員「なんとなく遠い」→ 区別がつかなくなる
```

次元削減は、この「スカスカ問題」を解消しながら、データの**本質的な構造（多様体）**を保ったまま低次元に落とし込む操作です。

---

## 直感的な仕組み——地球儀を地図に変換する例

UMAPの動きは「地球儀（3D）を世界地図（2D）に投影する」操作に似ています。

```
地球儀（3D）         →        世界地図（2D）
・球面上の位置                ・平面上の位置
・近い国は近くに               ・近い国はだいたい近くに
・完全には保存できない          ・歪みが生じる（グリーンランドが巨大に見えるアレ）
```

UMAPがやっていることも同じです：

1. **局所的な近傍グラフを構築**：各データ点の「近所」を探してグラフ（ネットワーク）を作る
2. **低次元でも同じグラフ構造を再現**：近所関係ができるだけ崩れないように2〜3次元へ配置する

重要なのは、**全体的な正確な距離ではなく、「誰が誰の隣人か」という近傍関係を保つ**ことに集中している点です。

---

## t-SNEとの違いと改良点

UMAPはt-SNEの後継として設計されており、いくつかの重要な改善があります。

| 比較項目 | t-SNE | UMAP |
|----------|-------|------|
| 計算速度 | 遅い（大規模データで特に） | **速い**（t-SNEの5〜10倍以上） |
| スケーラビリティ | 数万件が限界 | **数十万〜百万件も対応** |
| 大域的構造の保持 | 弱い（クラスター間の距離が不正確） | **比較的保持**（クラスター間の位置関係が意味を持つ） |
| 再現性 | `random_state` 固定でも変わることがある | **安定している** |
| 理論的背景 | KL divergence の最小化 | リーマン幾何学・位相的データ解析 |
| 新データへの適用 | 毎回全件から再計算が必要 | **`transform()` で新データを投影可能** |

### なぜ「大域的構造の保持」が重要なのか

t-SNEは「クラスター内のまとまり」はきれいに見せてくれますが、クラスター間の距離は恣意的になりがちです。UMAPでは、異なるクラスターの「離れ具合」にもある程度の意味があります。

## パラメータの説明

### `n_neighbors`（近傍数）——どのくらい「近所」を見るか

```python
umap.UMAP(n_neighbors=15)  # デフォルト：15
```

- **小さい値（5〜10）**：局所的な構造を優先。細かいクラスターが出やすいが、ノイズに敏感
- **大きい値（30〜50）**：大域的な構造を優先。全体のトポロジーを保ちやすい
- **偽情報クラスタリングでの目安**：15〜30（クラスター間の関係も見たいため）

> 感覚的には「k-NNのk」と同じ。「何人の隣人を考慮するか」

---

### `min_dist`（最小距離）——低次元での点の密集度

```python
umap.UMAP(min_dist=0.1)  # デフォルト：0.1
```

- **小さい値（0.0〜0.1）**：点が密集しやすい。クラスターが「塊」として見えやすい
- **大きい値（0.5〜1.0）**：点が散らばる。全体的な分布が見やすい
- **可視化目的なら**：0.0〜0.1 がおすすめ（クラスターの境界が明確になる）

---

### `metric`（距離指標）——「近さ」をどう測るか

```python
umap.UMAP(metric='cosine')  # テキスト埋め込みにはコサイン距離
```

| metric | 向いているデータ |
|--------|----------------|
| `euclidean` | 座標データ、画像の画素値など |
| `cosine` | テキスト埋め込み、TF-IDF |
| `manhattan` | スパースデータ |
| `correlation` | 時系列データ |

テキスト埋め込みには `cosine` を使うのが一般的です（後述）。

---

### `random_state`（乱数シード）——再現性の確保

```python
umap.UMAP(random_state=42)
```

UMAPは初期配置にランダム性があります。`random_state` を固定することで、同じデータから毎回同じ結果が得られます。論文・レポートを書く際は必ず固定してください。

## コサイン類似度とコサイン距離の違い

テキスト埋め込みでよく登場するので、しっかり理解しておきましょう。

### コサイン類似度（Cosine Similarity）

ベクトル同士の**向き（角度）**の一致度を測ります。大きさ（ノルム）は無視します。

```
cosine_similarity(A, B) = (A · B) / (|A| × |B|)

値の範囲：-1 〜 +1
  +1 : 同じ方向（意味が近い）
   0 : 直交（無関係）
  -1 : 逆方向（意味が反対）
```

### コサイン距離（Cosine Distance）

類似度を「距離」に変換したものです。

```
cosine_distance(A, B) = 1 - cosine_similarity(A, B)

値の範囲：0 〜 2
  0 : 同じ方向（距離ゼロ＝同一）
  1 : 直交
  2 : 逆方向（最大距離）
```

### なぜテキスト埋め込みにコサインを使うのか

テキスト埋め込みはベクトルの「向き」に意味が込められています。

```
「新型コロナ」の短い記事  →  小さなノルムのベクトル
「新型コロナ」の長い記事  →  大きなノルムのベクトル
```

ユークリッド距離はノルムの違いに引きずられてしまいますが、コサイン距離は向きだけを見るので、**記事の長さに左右されず「内容の近さ」を測れます**。

UMAPで `metric='cosine'` を指定すると、コサイン距離を使って近傍グラフを構築します。

## 今回の使い方——偽情報クラスタリングパイプラインでの役割

### パイプライン全体の流れ

```
生テキスト
    ↓ (1) 埋め込み生成
高次元ベクトル（例：1536次元）
    ↓ (2) UMAP で次元削減
低次元ベクトル（例：5〜15次元）
    ↓ (3) HDBSCANでクラスタリング
クラスターラベル
    ↓ (4) UMAP で可視化用に2D圧縮
2次元散布図
```

### UMAPを2段階使う理由

**1段階目：クラスタリング前の次元削減（15次元程度）**

```python
reducer_cluster = umap.UMAP(
    n_components=15,    # 15次元に削減
    n_neighbors=30,
    min_dist=0.0,       # クラスタリング用は密集させる
    metric='cosine',
    random_state=42
)
```

- 1536次元のままHDBSCANを動かすと「次元の呪い」でクラスタリング精度が落ちる
- 15次元程度に落とすことで、距離計算が意味を持ちやすくなる
- `n_components` は完全な2Dにはしない（構造を多少保ちたいため）

**2段階目：可視化用の2D圧縮**

```python
reducer_viz = umap.UMAP(
    n_components=2,     # 2次元に削減（可視化用）
    n_neighbors=15,
    min_dist=0.1,
    metric='cosine',
    random_state=42
)
```

- クラスタリング結果を人間が目で確認するための散布図を作る
- クラスターの「塊」と「分離」を視覚的に確認できる

### 偽情報分析での意義

- **近い位置 = 内容が近い**ので、クラスター内のテキストは「同じ偽情報ナラティブを共有している」と解釈できる
- **クラスター間の距離**も参考になる（完全に離れた偽情報トピック vs 部分的に重なるトピック）
- 新しい記事が来たとき `transform()` で既存の2D空間に投影できるため、**リアルタイム分析にも使いやすい**

## 実装の最小サンプル

```python
import umap
import numpy as np

# 埋め込みベクトル（例：100件 × 1536次元）
embeddings = np.load("embeddings.npy")

# クラスタリング用（15次元）
reducer = umap.UMAP(
    n_components=15,
    n_neighbors=30,
    min_dist=0.0,
    metric='cosine',
    random_state=42
)
reduced = reducer.fit_transform(embeddings)

# 可視化用（2次元）
reducer_2d = umap.UMAP(
    n_components=2,
    n_neighbors=15,
    min_dist=0.1,
    metric='cosine',
    random_state=42
)
coords_2d = reducer_2d.fit_transform(embeddings)
```

## まとめ

| 項目 | ポイント |
|------|----------|
| 何をするか | 高次元→低次元への次元削減 |
| なぜ必要か | 次元の呪いを避け、クラスタリング・可視化を可能にする |
| t-SNEとの違い | 速い・スケールする・大域構造を保つ・新データに適用可能 |
| テキストの距離指標 | `metric='cosine'`（内容の向きで近さを測る） |
| 今回の役割 | ①クラスタリング精度向上のための15D削減、②可視化のための2D削減 |
